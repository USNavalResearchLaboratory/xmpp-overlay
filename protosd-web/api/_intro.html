ProtoSD REST API

ProtoSD embeds a minimal Web server implementation based on Jersey's simple Web server implementation, which has been extended to do the following:

  1. HTTPS: for supporting secure connections
  2. Server side includes: for integrating files on the server side for convenience
  3. Mime support: for supporting several mime types, including image types (jpg, png, etc), css, javascript and php. With these extensions, the Web server can load in most Web content.
  4. Extended WADL Support: for generating dynamic WADL interfaces to be found at "/application.wadl" that include the javadoc comments from the Java source code..

The REST API uses this Web server implement and integrates the Jersey REST toolkit, which is an implementation of the JAX-RS (JSR 311) standard. Jersey essentially provided annotations extended from JAXB for the REST verbs required to create REST implementations. The use of annotations is interesting because it encourages stateless server side implementation, an essential feature for creating scalable REST applications.

ProtoSD High level Rest Functionality

ProtoSD supports the following functionality:

  1. Dynamic Service Discovery System (SDS) Instantiation: you can create, edit and delete instances of the JmDNS and INDI service discovery discovery systems using the API.
  2. Service Advertisements: ProtoSD supports the creations of new service for advertisement and provides support for all associated metadata required for creating complex service advertisements for the particular environment being deployed into. It also supports service deletion
  3. Service Discovery: all services are discovered according to the settings of the discovery system being used. The REST API provides multiple methods for searching the discovered services for fine tuning the results and providing rapid access to the the types of services you are interested in. There are also multiple levels of service representation for a high level view or a in depth look at all of the associated parameters.

Specifying The Content Type

You can specify the payload type and the return type for the messages by using the header's content-type and accept header:

  1. Content-type: by setting 'Content-Type': 'application/xml' you will be able to send XML to the POST or PUT operations and setting the 'Content-Type': 'application/json' would allow you to set a JSON payload.
  2. Accept: by setting 'Accept': 'application/xml' in the http header you tell the Web API to return XML and 'Accept': 'application/json' will force it to return JSON.
